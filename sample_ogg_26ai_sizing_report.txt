Parallel Replicat starting parameters
======================================
Use this section to start the target apply design. Validate with target database capacity,
transaction dependencies, constraints, indexes, triggers, and representative workload.
Oracle reference: https://docs.oracle.com/en/middleware/goldengate/core/21.3/coredoc/replicat-basic-parameters-parallel-replicat.html
- MAP_PARALLELISM starting point: 2 mapper thread(s). Default is 2; valid documented range is 1-100.
- Auto-tuned apply mode: use MIN_APPLY_PARALLELISM and MAX_APPLY_PARALLELISM together.
- MIN_APPLY_PARALLELISM starting point: 1
- MAX_APPLY_PARALLELISM starting point: 4
- Fixed apply mode alternative: APPLY_PARALLELISM starting point: 4. Use only if you do not use MIN/MAX apply parallelism.
- SPLIT_TRANS_RECS: leave disabled initially; consider only for large transactions after dependency and recovery testing.
- COMMIT_SERIALIZATION: use FULL only when target commit order must be forced; validate throughput impact.
- LOOK_AHEAD_TRANSACTIONS: keep the default starting point unless scheduling tests show a bottleneck.
- CHUNK_SIZE: keep the default starting point; increasing it can consume more Replicat memory.
- Tuning note: increase gradually while watching apply lag, CPU above 80 percent, target constraints, and transaction dependencies.

Validation requirements before production
=========================================
Validate this baseline with production-representative workload before go-live.
During validation, monitor Extract lag, Replicat lag, CPU utilization, CACHEMGR spill,
trail backlog growth, trail filesystem utilization, swap activity, and process restarts.
A generally healthy starting target is Extract lag below 5 seconds, Replicat lag below 10 seconds,
sustained CPU below 70 percent, trail filesystem below 70 percent, minimal CACHEMGR spill,
and no recurring resource-related GoldenGate abends.
